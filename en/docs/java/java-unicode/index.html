Unicode

Unicode俗稱萬國碼，基本上就是因應世界各國不同的文字，希望有一個統一的編碼，每一個數字都會對應到某個文字，比如中文unicode，你可以到 全字庫 中查詢，如果你查詢'中'這個字，可以看到會有對應的16進位的碼，這裏顯示的是4E2D

Java裏面的字元也是使用Unicode系統，我們在Java裏面可以測試看看。

char c ='中';
System.out.println(c + " 對應的碼" + (int)c );
System.out.println("轉成16位進："+Integer.toHexString(c));
－－－－－－－－－－－－
中 對應的碼20013
轉成16位進：4e2d

在程式中我們可以把單一的一個字直接轉成數字，會是20013(轉成16進位就會是4e2d)，這個數字我們稱為code point。

Unicode碼採用的編碼有很多種，常聽到的有UTF8、UTF-16、UTF-32，這些不同編碼指的是把原本相同的數字用不同方式儲存。最常見到的是UTF8，他會採取1-3bytes來儲存所對應的數字。UTF16是2 bytes，UTF32則是4 bytes。

以英文字大寫的A來做說明，在Ascii碼中，A代表的數字是65，在不同的編碼下，會儲存成不同的bytes

編碼佔用bytes16進位2進位
UTF814101000001
UTF16200 4100000000 01000001
UTF32400 00 00 4100000000 00000000 00000000 01000001
public static void main(String argv[]){
    char testCharacter ='A';
    byte[] bytes = ("" + testCharacter).getBytes(StandardCharsets.UTF_8);
    printBytes("UTF8", bytes);
    bytes = ("" + testCharacter).getBytes(StandardCharsets.UTF_16BE);
    printBytes("UTF16", bytes);
    try {
        bytes = ("" + testCharacter).getBytes("utf32");
        printBytes("utf32", bytes);
    } catch (UnsupportedEncodingException e) {
        throw new RuntimeException(e);
    }
}

private static void printBytes(String encoding, byte[] bytes) {
        System.out.printf("%s byte長度＝%d%n", encoding,bytes.length);
        for (byte b : bytes) {
            System.out.printf("%02x ", b);
        }
        System.out.println();
}
－－－－－－－－－－－－－－
UTF8 byte長度＝1
41 
UTF16 byte長度＝2
00 41 
utf32 byte長度＝4
00 00 00 41

你可以看到在儲存A這個字時，UTF8使用的byte數最少，UTF32則因為固定使用4 bytes，佔的儲存空間一定是最多。

我們再看另一個情況，如果我測試的字是中文字的'中'，

UTF8 byte長度＝3
e4 b8 ad 
UTF16 byte長度＝2
4e 2d 
utf32 byte長度＝4
00 00 4e 2d

會看到UTF8反而使用了3 bytes，但UTF16依舊是2 bytes。所以並不是所有的情況UTF8都一定是最省儲存空間的。

Why UTF16​

Java一開始設計時就想兼容Unicode，Unicode一開始只定義了0x0000~0xFFFF，最多6萬多字(65536)，於是採用了2 bytes來做為char的大小，但隨著字數越來越多原本的65536不夠用了，Unicode擴充後(1,112,064字)，在原本設計2 bytes的情況下，採用UTF16編碼是最適合的作法。這也是為什麼Java採用了UTF16而非UTF8的原因，Oracle官網有篇文章在說明這件事情，有興趣的可以參考看看。

在Java8之前，java.lang.String是使用char[]來表示文字，多個文字組合起來。每一個char都是2 bytes。

Java9後，則採用了Compact String，以 byte[]來儲存，在英文字採用1 bytes，有非ISO-8859-1以外的字才使用UTF16，以節省記憶體的空間。

Java 8 使用char[]
public final class String
    implements java.io.Serializable, Comparable<String>, CharSequence {
    private final char value[];
}
Java 9 使用byte[]
public final class String
  implements java.io.Serializable, Comparable<String>, CharSequence,
             Constable, ConstantDesc {
  private final byte[] value;
}
如果你用JDK17測試時要加上vm options: --add-opens java.base/java.lang=ALL-UNNAMED
否則沒有辦法存取private final field
String[] test = {"a","a中"};
try {
    Field value = String.class.getDeclaredField("value");
    value.setAccessible(true);
    Stream.of(test).forEach(s -> {
        try {
            byte[] bs = (byte[]) value.get(s);
            System.out.printf("%s,length = %d%n", s ,bs.length);
        } catch (IllegalAccessException e) {
            throw new RuntimeException(e);
        }
    });
} catch (NoSuchFieldException e) {
    throw new RuntimeException(e);
}
------------
a,length = 1
a中,length = 4

如果這個String裏面全部都是英文字，那可以大幅縮減記憶體使用量，每一個單字只使用了1 byte，但如果有任何一個字不是，則全部的字都會採用UTF16，每個單字就會需要2 bytes。

Unicode Supplementary

原有的0xFFFF不夠用後，Unicode 2.0開始了Supplmentary character的設計，Unicode 3.1後開始真的用到了所謂的supplementary部分，這些區域稱為Supplementary Multilingual Plane(原本的0x0000-0xFFFF稱為Basic Multilingual Plane)，簡單來說就是定義了更多的數字來對應所需要的文字。中文裏面也有些字是建在Supplementary character中，但這些字對應的code point都會大於0xFFFF(65536)，在UTF16處理上會採用兩個UTF16字來表示一個字，也就是這些字一次至少會佔到4 bytes，當然這些區域的字也就無法以單一的char來使用。Java也在1.5才正式支援Supplementary Multilingual Plane中的字。每個Java版本支援的Unicode version也都不儘相同。

舉個位於supplementary中的字當例子：𨳎 （這個字在網頁顯示上不知道為什麼Safari顯示怪怪的,Chome到是沒什麼問題)

如果我試著在Java中使用以下程式會發生問題

char testCharacter ='𨳎'; //compile無法過
String testSMP = "𨳎";

因為𨳎實際上在UTF16中是兩個字組起來，如果你直接將這個字貼到IntellJ中，他會自動變成兩個unicode的字。

String smp = "\uD863\uDCCE";
byte[] bytesOfSMP = smp.getBytes(StandardCharsets.UTF_16BE);
System.out.println(smp+" 長度＝" + smp.length());
System.out.println(smp+" bytes 長度＝" + bytesOfSMP.length);
for (byte b : bytesOfSMP) {
    System.out.printf("%02x ",b);
}
------------------
𨳎 長度＝2
𨳎 bytes 長度＝4
d8 63 dc ce

可以看到雖然看起來字型是一個字，但實際上他佔了兩個字的空間。

這個現況除了定義在SMP中的字外，SMP還有不少的表情符號。

Emoji​

比如🇹🇼，看起來是一個字，但實際上是2個字組合起來，而且這兩個字都是supplementary中的字，如果我們用程式把他拆開，你會看到總共長度是4，佔了8 bytes。

String tw = "🇹🇼";
System.out.println("length = "+tw.length());
System.out.println("bytes length = "+tw.getBytes(StandardCharsets.UTF_16BE).length);
char[] chars = tw.toCharArray();
for (char c : chars) {
    System.out.print(String.format("%04x ", (int) c));
}
-----------
length = 4
bytes length = 8
d83c ddf9 d83c ddfc

因為現在看到的一個字事實上可能是多個字組合起來的，所以Java String中有提供codePoints()讓你讀取裏面的每一個組成的unicode字。

String tw = "🇹🇼";
tw.codePoints().forEach(codePoint -> {
    System.out.println("codePoint = " + codePoint);
    System.out.println(Character.toChars(codePoint));//把unicode codepoint 轉成文字
});
---------------
codePoint = 127481
🇹
codePoint = 127484
🇼

裏面總共有2個unicode code的字組合起來，第一個是🇹第二個是🇼
